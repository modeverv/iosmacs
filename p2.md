# Emacs-following terminal wait plan

## Goal

`M-x tetris` のような timer/redisplay driven な UI を滑らかにするため、`iosmacs` を terminal widget 主導ではなく Emacs event loop の従者として動かす。

この方針では、Swift/xterm.js は入力・resize・出力転送の薄い bridge に留める。待機の主導権は Emacs C core に置き、`iosmacs` host facade は Emacs が指定した deadline までだけ blocking wait する。

## Current issue

現状の direct tty path は `wasmacs` に近づいたが、`iosmacs_host_wait_for_input()` が固定 50ms wait で、さらに `wait_reading_process_output` を一部置き換えている。

これは通常入力では動くが、Emacs が内部 timer のために「この時刻に起きたい」と決めている場面で host 側が粗い cadence を作りやすい。`tetris` のような package では、入力 readiness だけでなく timer, redisplay, process wait の本来経路を残す必要がある。

## Target shape

```text
Swift/xterm.js
  -> input/resize を C facade に notify

Emacs C core
  -> 次の event/timer deadline を決める
  -> iosmacs_host_wait_for_input(timeout_ms)
  -> timeout/input/resize の理由で戻る
  -> Emacs 本来の timer/redisplay/process loop を続ける
```

`iosmacs` は「いつ起きるか」を決めない。Emacs が決めた timeout に従う。

## Return codes

`iosmacs_host_wait_for_input(timeout_ms)` と `iosmacs_os_terminal_wait_for_input(timeout_ms)` は次を返す。

```text
0 = timeout / scheduler wake
1 = input
2 = resize
-1 = unavailable / unsupported
```

今の C facade はすでに `0/1/2` を返せるため、まずは public signature と呼び出し側を timeout-aware にする。

実装では同じ値を `IOSMACS_HOST_WAIT_TIMEOUT`, `IOSMACS_HOST_WAIT_INPUT`, `IOSMACS_HOST_WAIT_RESIZE`, `IOSMACS_HOST_WAIT_UNAVAILABLE` として C header にも出す。

## Implementation steps

1. `iosmacs_host_wait_for_input(void)` を `iosmacs_host_wait_for_input(int timeout_ms)` に変更する。

2. `iosmacs_os_terminal_wait_for_input(int timeout_ms)` は現在の pthread condition variable 実装を維持する。

   - `timeout_ms < 0`: indefinite wait
   - `timeout_ms == 0`: poll相当
   - `timeout_ms > 0`: timed wait

3. `sysdep.c` patch の direct tty read path は、byte がまだ無い時だけ短い bounded wait を使う。

   ```text
   iosmacs_host_wait_for_input(50)
   ```

   ここは fd read の互換 path なので、timer ownership は主に `keyboard.c` 側へ寄せる。

4. `keyboard.c` patch に `end_time` から millisecond timeout を計算する helper を入れる。

   ```text
   end_time == NULL -> 50ms
   end_time <= now  -> 0ms
   end_time > now   -> ceil((end_time - now) in ms)
   ```

5. `read_decoded_event_from_main_queue` の直前 waitpoint は `end_time` を見て timeout を渡す。

   ```text
   iosmacs_host_wait_for_input(timeout_ms)
   ```

6. `wait_reading_process_output` の丸ごと置き換えはやめる。

   Emacs の既存処理は、`end_time` がある時に `wait_reading_process_output(sec, nsec, ...)` へ正しい duration を渡し、`end_time` が無い時も redisplay 可否を見て待つ。ここを残すことで timer/process/redisplay の本来経路を保つ。

7. それでも `tetris` などで cadence が粗い場合は、direct facade だけで粘らず fake tty を fd-readable な pipe/socketpair 主体に戻す。

   ```text
   Swift/xterm input
     -> socketpair/pipe peer に write
     -> Emacs stdin fd が readable
     -> wait_reading_process_output/select が自然に起きる
   ```

   これは Emacs の既存 event loop に最も従う形だが、iOS の tty互換 shim と resize/output path の整備が必要になるため second step とする。

## Done criteria for this phase

- C host facade public API が timeout-aware になる。
- copied Emacs source patch が timeout-aware waitpoint を生成する。
- copied Emacs source patch が `wait_reading_process_output` を置き換えない。
- 実機テストはこの段階では不要。
